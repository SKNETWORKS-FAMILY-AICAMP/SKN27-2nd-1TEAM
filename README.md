## 프로젝트 개요

통신사 고객 데이터를 기반으로 이탈 가능성이 높은 고객을 사전에 예측하고, 팀별 맞춤 대시보드를 통해 실질적인 비즈니스 액션으로 연결하는 AI 서비스입니다.

단순한 예측 모델에 그치지 않고, **분석 · 마케팅 · 영업CS · 운영** 네 팀의 실무 시나리오를 반영한 화면을 각각 구성하여 이탈 예측 결과가 실제 의사결정에 활용될 수 있도록 설계했습니다.

---

## 배경 및 필요성

고객 이탈은 통신사 매출에 직접적인 타격을 줍니다. 신규 고객 유치 비용은 기존 고객 유지 비용의 5배 이상으로 알려져 있으며, 이탈 후 대응은 사실상 효과가 없습니다.

이 서비스는 **이탈 전에 위험 고객을 식별**하여 각 부서가 선제적으로 대응할 수 있도록 돕는 것을 목표로 합니다.

---

## 프로젝트 목표

1. **이탈 위험 고객 사전 탐지** → AI 예측 모델(AUC 0.85)로 이탈 확률 실시간 산출
2. **데이터 누수 없는 정직한 모델링** → Churn Score, Churn Reason, CLTV 등 사후 변수 철저 제거
3. **통계적으로 검증된 EDA** → 카이제곱 검정, T-test, 생존 분석으로 인사이트 수치 증명
4. **팀별 맞춤 서비스** → 분석팀·마케팅팀·영업팀·운영팀 각자의 역할에 최적화된 화면 제공

## Step 2. 인코딩 및 스케일링 (Encoding & Scaling)

모델의 과적합을 막고 평가의 무결성을 담보하기 위해 철저히 통제된 순서로 변환을 수행했습니다.

#### 1. 데이터 누수(Data Leakage) 원천 차단

* **제거 대상:** `Churn Reason`, `Churn Label`, `CustomerID`, `Count`, `Country`, `State`, `City`, `Zip Code`, `Latitude`, `Longitude`, `Lat Long`, `Churn Score`
* **통제 사유:** 모델에 정답 정보가 미리 유입되거나 식별자로 인해 불필요한 노이즈가 발생하는 것을 시스템적으로 사전 차단했습니다.

#### 2. 훈련/테스트 셋 분할 (Train/Test Split)

* **설정값:** 80:20 비율 분할, `stratify=y` (층화 추출), `random_state=42`
* **통제 사유:** 타겟 클래스의 불균형 상태를 일정하게 유지하며, 데이터 누수를 막기 위해 반드시 인코딩과 스케일링 **이전**에 물리적인 데이터 분할을 선행했습니다.

#### 3. 타겟 인코딩 (Target Encoding)

* **적용 방식:** `TargetEncoder(smooth="auto")` 활용, 수치형 변수는 `remainder='passthrough'` 옵션을 통해 통과시켰습니다.
* **통제 사유:** 범주형(Categorical) 변수에만 타겟 인코딩을 적용하며, 내부 교차 검증 로직이 포함된 Scikit-learn 내장 모듈을 사용하여 타겟 인코딩 특유의 과적합 현상을 방어했습니다.

#### 4. 표준화 스케일링 (Standard Scaling)

* **적용 방식:** `StandardScaler()` 활용.
* **통제 사유:** 모델의 학습 안정성을 확보하기 위해 모든 피처를 평균 0, 표준편차 1로 정규화했습니다. 데이터 무결성 유지를 위해 훈련 데이터(`X_train`)에만 `fit_transform`을 수행하고, 테스트 데이터(`X_test`)에는 훈련 데이터의 분포 기준을 엄격히 종속시켜 `transform`만 적용했습니다.

## 모델링

본 프로젝트는 단일 알고리즘의 편향을 막고 일반화 성능을 극대화하기 위하여, 3개의 베이스 모델(Base Model)과 1개의 메타 모델(Meta Model)을 결합한 스태킹 구조를 채택했습니다.

| 모델명                            | 핵심 역할                            | 채택 이유 및 장점                                                                                                   | 한계점 및 통제 방안                                                                                                         |
| :-------------------------------- | :----------------------------------- | :------------------------------------------------------------------------------------------------------------------ | :-------------------------------------------------------------------------------------------------------------------------- |
| LightGBM`<br>`(Base)            | '빠른 속도와 예리한 패턴 탐지'       | 40개가 넘는 파생 변수 사이에서 이탈에 영향을 주는 핵심 변수를 가장 빠르고 예리하게 찾아냈습니다.                    | 깊게 학습하다 과적합될 위험이 있어, 트리의 깊이와 학습률을 엄격하게 제한하여 통제했습니다.                                  |
| XGBoost`<br>`(Base)             | '묵직한 안정성 유지'                 | 비선형적인 데이터 패턴을 단단하게 학습하여, LightGBM이 놓칠 수 있는 데이터의 균형을 수학적으로 잡아줍니다.          | 학습 속도가 무겁기 때문에, 데이터를 샘플링해서 가볍게 학습하도록 규제(Regularization)를 걸었습니다.                         |
| CatBoost`<br>`(Base)            | '문자형(범주형) 데이터 스페셜리스트' | 계약 형태나 요금제 같은 문자형 데이터를 변환 손실 없이 완벽하게 처리하며, 자체적인 과적합 방어력이 최고 수준입니다. | 특정 파이썬 패키지 버전에 종속되는 경향이 있어, 시스템 환경(requirements)을 단일 버전으로 깐깐하게 고정했습니다.            |
| Logistic Regression`<br>`(Meta) | '최종 결정권자'                      | 앞선 3명의 AI(Base)가 각자의 방식으로 예측한 확률(0~1)을 전달받아 종합적으로 평가하고 최종 결론을 내렸습니다.       | 복잡한 분석은 3명의 Base가 이미 끝냈으므로, 앙상블의 과적합을 막기 위해 가장 단순하고 투명한 선형 모델을 리더로 세웠습니다. |
| 5-Fold CV`<br>`(검증 방식)      | '우연의 일치(운) 배제'               | 전체 데이터를 5조각으로 나누어 번갈아 가며 학습하고 시험을 치게 하였습니다.                                         | 특정 데이터 셋에서만 점수가 높게 나오는 거품을 빼고, 실전 환경에서도 동일한 예측력이 나오는지 교차 검증했습니다.            |

각기 다른 강점을 가진 lgbm, xgb, catboost가 데이터를 다각도로 분석하여 교차 검증(5-fold)을 거친 의견을 내고, logistic regression이 이를 종합하여 가장 안전한 방법이라고 생각해서 사용했습니다

## 하이퍼 파라미터

#### 1. LightGBM

| 파라미터명        | 탐색 범위         | 통제 목적                                   |
| :---------------- | :---------------- | :------------------------------------------ |
| n_estimators      | 100 ~ 500         | 부스팅 트리의 총 개수 제한 (과적합 방어)    |
| learning_rate     | 0.01 ~ 0.1 (log)  | 학습 보폭 미세 조정                         |
| num_leaves        | 20 ~ 150          | 단일 트리의 최대 잎 노드 수 제한            |
| max_depth         | 3 ~ 10            | 트리의 최대 깊이 물리적 제한                |
| colsample_bytree  | 0.5 ~ 1.0         | 다중공선성 완화를 위한 피처 샘플링 비율     |
| subsample         | 0.5 ~ 1.0         | 특정 데이터 편향 방지를 위한 데이터 샘플링  |
| min_child_samples | 10 ~ 100          | 소수 이상치(Outlier) 패턴화 차단            |
| reg_alpha         | 1e-4 ~ 10.0 (log) | L1 페널티 (불필요한 피처 가중치 0으로 수렴) |
| reg_lambda        | 1e-4 ~ 10.0 (log) | L2 페널티 (가중치 폭주 현상 억제)           |

#### 2. XGBoost

| 파라미터명       | 탐색 범위         | 통제 목적                                          |
| :--------------- | :---------------- | :------------------------------------------------- |
| n_estimators     | 100 ~ 500         | 부스팅 트리의 총 개수 고정 (연산 낭비 최소화)      |
| learning_rate    | 0.01 ~ 0.1 (log)  | 학습 속도의 미세 조정                              |
| max_depth        | 3 ~ 10            | 트리 구조의 과도한 복잡화 방지                     |
| colsample_bytree | 0.5 ~ 1.0         | 변수 종속성 분산을 통한 모델 강건성 확보           |
| subsample        | 0.5 ~ 1.0         | 훈련 데이터 전체 암기(과적합) 방어                 |
| min_child_weight | 1 ~ 20            | 지엽적 데이터 분할 차단                            |
| gamma            | 0.0 ~ 5.0         | 노드 분할에 대한 보수적 접근 (과적합 방지)         |
| reg_alpha        | 1e-4 ~ 10.0 (log) | L1 규제 (예측력 없는 피처 제거)                    |
| reg_lambda       | 1e-4 ~ 10.0 (log) | L2 규제 (가중치 폭주 제어)                         |
| scale_pos_weight | ratio (고정값)    | 소수 클래스(이탈)에 대한 가중치 부여 (불균형 보정) |

#### 3. CatBoost

| 파라미터명          | 탐색 범위         | 통제 목적                                    |
| :------------------ | :---------------- | :------------------------------------------- |
| iterations          | 100 ~ 500         | 총 반복 훈련 횟수 제한 (과적합 및 시간 통제) |
| learning_rate       | 0.01 ~ 0.1 (log)  | 가중치 업데이트 속도 미세 조절               |
| depth               | 3 ~ 10            | 대칭 트리(Symmetric Tree) 깊이 제한          |
| l2_leaf_reg         | 1e-4 ~ 10.0 (log) | 리프 노드 가중치 폭주 억제 (L2 규제)         |
| bagging_temperature | 0.0 ~ 1.0         | 베이지안 부트스트랩 노이즈 강도 조절         |
| border_count        | 32 ~ 255          | 연속형 변수 분할(Binning) 경계선 수 탐색     |
| auto_class_weights  | "Balanced" (고정) | 클래스 불균형에 따른 타겟 가중치 자동 재조정 |

이번 프로젝트에서는 머신러닝 전반적인 프로세스를 직접 경험해볼 수 있어 뜻깊었습니다. 각자 머신러닝 모델링을 제작하여 비교해보는 작업을 했기 때문에 CatBoost 등 다양한 모델을 활용한 앙상블과 Voting 기법을 시도하며 여러 방향으로 성능을 개선하는 과정이 재미있고 학습에 도움이 되었다고 생각합니다. 또한 ERD 설계를 통해 데이터베이스를 직접 구성하고 SQL 연결까지 경험하면서 데이터 관리에 대한 이해도를 높일 수 있었습니다. 단순한 모델 구현을 넘어 데이터 설계부터 모델링까지 아우르는 경험을 쌓을 수 있었던 프로젝트였습니다..

**주연중**:
프로젝트 하면서 불편한점 하나없었습니다. 모두 열심히 참여하신거 같습니다. 저도 이번에는 모델링등에 참여하며 많이 배우는 뜻깊은 시간이 되었던거 같습니다. 이번 경험을 발판삼아 앞으로 더 발전하는 사람이 되겠습니다. 이탈에 일조 모두 앞으로 화이팅!

## [기술 파트 ② — 인코딩 & 스케일링]


모델의 과적합을 막고 평가의 무결성을 지키기 위해 **철저히 통제된 순서**로 전처리를 수행했습니다.

- **① 선 분할 (80:20, stratify=y)** — 인코딩·스케일링 이전에 데이터를 먼저 쪼갭니다. 순서가 바뀌면 테스트 데이터 정보가 훈련에 스며드는 누수가 발생하기 때문입니다.
- **② Target Encoding** — `TargetEncoder(smooth="auto")` 적용. 내부 교차 검증이 내장된 Scikit-learn 모듈을 사용해 범주형 변수의 과적합 현상을 자동 방어했습니다.
- **③ Standard Scaling** — 훈련 데이터에만 `fit_transform`, 테스트 데이터에는 `transform`만 적용해 훈련 분포 기준을 엄격히 종속시켰습니다.

## [기술 파트 ③ — AI 모델: 스태킹 앙상블]


저희는 단일 AI의 편향을 막기 위해, 각기 다른 강점을 가진 AI들을 모아 **'전문가 위원회'** 를 구성했습니다.  
기술적으로는 **스태킹(Stacking) 앙상블** 기법입니다.

---

### Base Model ① — LightGBM : 빠른 속도와 예리한 패턴 탐지


47개 파생 피처 사이에서 이탈에 영향을 주는 핵심 변수를 가장 빠르고 예리하게 찾아냅니다.  
단, 깊게 학습하다 과적합될 위험이 있어 **Optuna로 아래 범위를 50회 탐색**해 최적값을 찾았습니다.

| 파라미터 | 탐색 범위 | 통제 목적 |
|---|---|---|
| `n_estimators` | 100 ~ 500 | 부스팅 트리 수 제한 (과적합 방어) |
| `learning_rate` | 0.01 ~ 0.1 (log) | 학습 보폭 미세 조정 |
| `num_leaves` | 20 ~ 150 | 단일 트리 복잡도 제한 |
| `reg_alpha / reg_lambda` | 1e-4 ~ 10.0 (log) | L1·L2 규제로 가중치 폭주 억제 |

> **Best AUC (5-fold CV): 0.8645**

---

### Base Model ② — XGBoost : 묵직한 안정성


비선형적인 데이터 패턴을 단단하게 학습하여, LightGBM이 놓칠 수 있는 균형을 수학적으로 잡아줍니다.  
클래스 불균형 보정을 위해 `scale_pos_weight`를 **실제 유지/이탈 비율**로 고정했습니다.

| 파라미터 | 탐색 범위 | 통제 목적 |
|---|---|---|
| `max_depth` | 3 ~ 10 | 트리의 과도한 복잡화 방지 |
| `gamma` | 0.0 ~ 5.0 | 노드 분할에 대한 보수적 접근 |
| `min_child_weight` | 1 ~ 20 | 지엽적 데이터 분할 차단 |
| `scale_pos_weight` | 고정 (이탈 비율) | 소수 클래스(이탈)에 가중치 부여 |

> **Best AUC (5-fold CV): 0.8650**

---

### Base Model ③ — CatBoost : 범주형 데이터 스페셜리스트


계약 형태·결제 방식 같은 문자형 데이터를 변환 손실 없이 완벽하게 처리합니다.  
세 모델 중 자체 과적합 방어력이 가장 높으며, `auto_class_weights="Balanced"` 옵션으로 불균형까지 자동 보정됩니다.

| 파라미터 | 탐색 범위 | 통제 목적 |
|---|---|---|
| `depth` | 3 ~ 10 | 대칭 트리 깊이 제한 |
| `l2_leaf_reg` | 1e-4 ~ 10.0 (log) | 리프 노드 가중치 폭주 억제 |
| `bagging_temperature` | 0.0 ~ 1.0 | 베이지안 부트스트랩 노이즈 강도 조절 |
| `auto_class_weights` | "Balanced" (고정) | 클래스 불균형 자동 보정 |

> **Best AUC (5-fold CV): 0.8657**

---

### Meta Model — Logistic Regression : 최종 결정권자

앞선 3개의 Base 모델이 각자의 방식으로 예측한 확률(0~1)을 전달받아 **종합적으로 평가하고 최종 결론**을 냅니다.  
복잡한 분석은 Base 모델들이 이미 끝냈으므로, 앙상블의 과적합을 막기 위해 **가장 단순하고 투명한 선형 모델**을 메타 리더로 세웠습니다. (`C=0.05` 강한 규제 적용)

---

### 최종 성능 지표

| 지표 | 값 |
|---|---|
| **Test AUC** | **0.8512** |
| Accuracy | 80.2% |
| 이탈 Recall (TP율) | **78.1%** — 10명 중 7~8명 사전 감지 |
| 유지 Recall (TN율) | 76.9% |

## [비즈니스 파트: 부서별 맞춤형 대시보드]

---

### ① 운영팀 — 시스템 전체를 관리한다


운영팀은 전사 고객 데이터와 시스템 전반을 관리하는 **컨트롤 타워**입니다.

- **메인 대시보드** — 이탈 위험 현황 한눈에 파악 (🟢 안전 / 🟡 주의 / 🔴 고위험 3등급)
- **일괄 예측** — CSV 업로드 또는 DB 테이블 선택만으로 수천 명의 위험도 일괄 산출
- **자동 리포트** — 주간·월간 이탈 현황 리포트 자동 생성 + CSV 다운로드
- **고객 등록/수정** — 신규 고객 ID 형식 검증부터 기존 고객 정보 수정까지 데이터 무결성 유지
- **예측 이력 조회** — DB에 누적된 예측 결과를 필터링·검색·다운로드

---

### ② 분석팀 — 모델을 검증하고 고객을 세분화한다


분석팀의 역할은 AI 모델의 신뢰도를 검증하고, 고객을 의미 있는 그룹으로 분류하는 것입니다.

- **모델 성능 지표 페이지** — AUC · Accuracy · 혼동 행렬 · 변수 중요도를 현업 담당자에게 투명하게 공개. 모델이 신뢰할 수 있는 수준인지 지속 모니터링
- **K-Means 세그먼트** — 군집 수 2~6개를 직접 선택. 이탈률·평균 월요금·이용 기간 기준으로 자동 그룹화 후 아래 기준으로 명명:

| 세그먼트 기준 | 자동 명칭 |
|---|---|
| 이탈률 ≥ 40% | 🔴 고위험 |
| 평균 이용기간 ≥ 40개월 | 🟢 충성 |
| 평균 월요금 ≥ $75 | 🟡 고가치 |
| 그 외 | 🔵 일반 |
